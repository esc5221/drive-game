분류drive-game · 네트워크

1.0요약

링크로 초대한 친구들이 같은 트랙에서 서로의 차를 실시간으로 보고, 카운트다운 동시 출발 → 결과 보드로 한판씩 겨루는 프렌즈 룸 고스트 레이싱을 구현한다. 서버는 Cloudflare Durable Object 방 하나당 하나가 바이너리 상태 패킷과 채팅·레이스 이벤트를 릴레이만 하고, 물리·충돌·판정은 전부 클라이언트에 남는다. 퍼블릭 로비는 만들지 않는다 — 방은 링크/코드로만 존재한다.

Decision · 결정 서버 권위 물리 없음 · 충돌 없음 · 순수 릴레이. 이 게임의 본질은 타임어택(베스트랩·고스트·아이디얼랩)이고, 240 Hz 커스텀 물리를 서버로 옮기는 순간 조작감과 비용 둘 다 잃는다. 원격 차는 "실시간 스트리밍되는 고스트"로 취급한다 — 기존 ghost.js(10 Hz 위치+쿼터니언 기록·보간 재생)가 이미 그 렌더러다.
Constraint · 비용 불변식 멀티가 실제로 일어나는 순간에만 과금이 발생해야 한다. OFF면 코드도 안 싣고, 혼자면 침묵하고, 방이 비면 hibernate. 상세 설계는 6.0.

2.0모드 결정 — 왜 고스트 레이싱인가

옵션조작감구현 비용판정
서버 권위 + 롤백>50 ms RTT에서 되감기 체감, 240 Hz 물리 서버 포팅 필요매우 큼 (물리 이식 + 예측/보정)기각 — 브라우저 게임에 과설계
P2P (WebRTC 메시)좋음 (직결)큼 (시그널링·NAT·N² 연결)기각 — 모바일 NAT 신뢰 불가
고스트 릴레이 (DO)내 차는 로컬 240 Hz 그대로 = 완벽. 남의 차만 보간작음 (기존 고스트 시스템 재활용)채택 — 게임 본질(타임어택)과 정합

충돌이 없으므로 롤백·서버 재조정·치팅 방지 물리가 전부 불필요해진다 — 이 선택이 이후 모든 단순함의 원천이다.

4.0지연 보정 — 원격 차 재생 파이프라인

서버 권위가 없으므로 문제는 하나로 압축된다: 남의 차를 얼마나 부드럽고 정확하게 그리느냐. 플레이어별 독립 파이프라인:

① 적응형 지터 버퍼   재생 지연 = max(120 ms, 2×관측 지터). 스냅샷 2~3개 뒤에서 재생
② Hermite 보간       pos+vel 3차 보간 — 200 km/h면 틱당 4.6 m: 선형은 코너를 깎는다
③ quaternion slerp   자세 보간
④ dead reckoning     버퍼 고갈 시 마지막 vel+yaw rate로 최대 250 ms 외삽 → 페이드아웃
⑤ 수렴 스냅          새 패킷 오차 <0.5 m → 부드럽게 blend / 크면 순간이동(리스폰 판정)
  • 시계 동기화 불필요 — 스트림별 상대시간 재생(지터 버퍼)이라 클라 시계 오프셋이 무관하다.
  • DO 배치 — 첫 접속자 근처 PoP에 뜬다(한국 유저끼리면 ICN/NRT, RTT 10~40 ms). 어차피 보간 지연 120 ms가 지배적이라 고스트 레이싱엔 충분.

5.0IA · 유저플로우 — 링크 전용

방은 링크/코드로만 존재한다. URL이 곧 초대장 — 카톡에 링크 하나 던지면 끝이고, 코드 입력은 백업 수단이다. 퍼블릭 로비를 없앤 결과 채팅 모더레이션 문제가 사라지고(신뢰 그룹), 방이 의도적으로 만들 때만 존재해 비용도 더 준다.

5.1  채팅 — 되고, 링크 전용이라 안전하다

같은 WS 릴레이에 JSON chat 메시지 한 종류 추가라 서버 비용·복잡도는 사실상 0(저빈도, 20:1 카운트). 관건은 입력 UX:

데스크톱   Enter = 채팅 입력 열기 → 입력 중 게임 키 입력 차단(input gate) → Enter 전송 · Esc 닫기
모바일     주행 중 타이핑 불가 → 일시정지/결과 화면에서 입력 + 퀵챗 4버튼("GG" "ㅋㅋ" "고고" "한판 더")
표시       좌하단 피드, 최근 5줄, 8 s 후 페이드아웃 · 닉 색상 = 차 틴트 색과 동일
안전       링크 초대 방 = 신뢰 그룹 → 필터/신고 불요 (퍼블릭이었다면 불가능했던 결정)

5.2  레이스 세션 — 시작/종료 UX

프렌즈 방이므로 "한판" 구조가 핵심이 된다. 방 상태머신:

FREE(자유주행) ──호스트 RACE──▶ COUNTDOWN(5 s) ──▶ RACING ──▶ RESULTS ──▶ FREE
                                                              (전원 완주 or 타임아웃)
  • 시작 — 호스트만 RACE 버튼(랩 수 1/3/5 선택). 서버가 race_start 브로드캐스트 → 전원 차를 그리드로 리셋(출발 직선에 2열 지그재그 스폰, 겹침 없음) → 입력 잠금.
  • 카운트다운 오버레이 — 화면 중앙 대형 3 · 2 · 1 · GO + 비프음(기존 WebAudio로 오실레이터 3단 비프, GO는 높은 톤). 시계 동기화는 수신시각 + RTT/2 보정으로 충분 — 충돌이 없으니 ±50 ms 스큐는 각자 GO부터 랩을 재므로 공정성에 영향 없음.
  • 주행 중 HUD — 타이밍 옆에 P2/4 · LAP 1/3. 순위는 로컬 계산(완주 랩 수 + trackS 진행도, 원격 상태로부터) — 서버 판정 불요.
  • 종료 — 각자 N랩 완주 시 finish {totalMs} 송신, 서버가 도착 순서대로 브로드캐스트. 결과 오버레이: 다크 카드에 순위 보드(1st 닉 — 타임, 갭 +x.xx, 베스트랩 하이라이트), 내 행 강조. 하단 버튼 [한판 더](호스트)·[자유주행]. 미완주자는 우승자 완주 후 120 s 타임아웃 → DNF. 아직 달리는 사람 화면엔 확정 순위가 실시간으로 쌓임.
  • 난입 — RACING 중 입장한 게스트는 자유주행(관전 태그), 다음 판부터 참가.
안 만드는 것근거
퍼블릭 로비 · 방 목록링크 전용으로 단순화 — 채팅 안전·비용·IA 전부 이득

6.0DO 비용 설계 — "멀티일 때만 과금"

Cloudflare 과금 모델의 세 축(요청·듀레이션·스토리지)을 각각 0으로 눌러놓고, 실제 동시 주행이 일어나는 순간에만 비용이 켜지게 설계한다.

6.1  과금 모델과 대응 설계

과금 축규칙대응 설계
요청 (수신 WS)수신 메시지 20:1로 카운트, 발신은 무료, 프로토콜 ping 무료서버 틱 없이 수신 즉시 릴레이 — 발신이 공짜라 배칭보다 릴레이가 더 싸고 단순
듀레이션 (메모리 상주)DO가 깨어 있는 동안만 GB-s 과금Hibernation API: acceptWebSocket + serializeAttachment(로스터 메타) + setWebSocketAutoResponse(ping/pong이 DO를 깨우지 않음). alarm/setInterval 금지 — 서버 주기 루프는 DO를 상시 깨워 듀레이션을 태운다
스토리지 (SQLite)2026-01부터 과금저장 안 함 — 로스터·세션 베스트랩은 인메모리+소켓 attachment. 영속 랭킹은 스코프 밖

6.2  클라이언트 게이팅 — 비용이 켜지는 조건을 좁힌다

# 위에서 아래로 갈수록 비용이 커진다. 각 단계는 명시적 조건으로만 진입.
ONLINE=OFF        네트워크 모듈 dynamic import조차 안 함              비용 0
메뉴/일시정지      연결 유지하되 송신 0 Hz, 60 s 지나면 disconnect      ≈ 0
방에 혼자        roster<2 → 상태 송신 0 Hz (join 브로드캐스트만 대기)  hibernate → 0
탭 숨김           visibilitychange → 송신 정지, 30 s 후 disconnect     ≈ 0
동시 주행 (진짜 멀티) 12 Hz 송신 · 즉시 릴레이                          과금 구간
핵심 아이디어 "방에 혼자면 침묵"이 최대 절감 장치다. 호스트가 방을 만들고 친구를 기다리는 동안 송신이 0이면 DO에 들어오는 메시지가 0 → hibernate → 요청·듀레이션 모두 0. 게스트의 join 브로드캐스트(그 순간만 깸)를 받으면 즉시 송신을 재개한다. 링크 전용이라 방 자체가 의도적으로 만들 때만 존재하므로, 결과적으로 친구들이 실제로 같이 달리는 시간에만 과금된다.
